# TON Agentic Wallet

## Overview
В данном репозитории реализованы смарт-контракты для TON Agentic Wallets — кошельков, созданных для управления AI-агентами. Агенты могут совершать любые транзакции в блокчейне TON (используя TON MCP Server).

Референсы: [Wallet V5](https://github.com/ton-blockchain/tolk-bench/tree/master/contracts_Tolk/05_wallet-v5), [NFT](https://github.com/ton-blockchain/tolk-bench/tree/master/contracts_Tolk/02_nft)

## Деплой
Деплой происходит через единый контракт NFT-коллекции со всеми agentic wallets. Для создания кошелька пользователь должен подписать данные хранилища контракта (комбинацию master_address, agent_public_key, subwallet_id, master_public_key). Исходя из этих данных формируется уникальный индекс NFT. 

**Плюсы такого подхода:**
- Создание кошелька происходит при первом депозите — не требуется лишнее действие со стороны юзера. Создание кошелька возможно как при депозите TON, так и жетонов.
- При изначальной настройке кошелька агент может запросить у юзера подписи сразу для нескольких subwallet_id, чтобы в дальнейшем деплой кошельков происходил без участия юзера. 
- "Спам" кошельками невозможен — для деплоя корректного кошелька необходима подпись юзера.
- Для индексации достаточно смотреть транзакции на одном контракте

**Ограничения:**
- В теории, при деплое можно указать master_public_key не соответствующий кошельку пользователя, поэтому при отображении agentic wallets необходимо проверять соответствие master_public_key и master_address.
- Сейчас при подписи на деплое не используется collection address, поэтому при нескольких NFT коллекциях можно будет деплоить кошельки с одинаковыми данными с помощью одинаковых подписей. 

## Управление кошельком
- Транзакции с кошелька отправляются так же, как с обычного Wallet V5, но подписывать сообщения можно двумя способами: с помощью ключа агента, и с помощью ключа юзера.
- Юзер может деактивировать агента через external сообщения и по средствам отправки транзакции с текстовым комментарием "deactivate"
- Юзер может передать управление другому агенту (заменить agent_public_key)
  
## Лимиты агента
На данный момент не используется никаких лимитов на транзакции для агентов ("Fund what you risk"). Это осознонное решение для упрощения первой версии контрактов.

## P.S.
Для развертывания первого agentic-кошелька юзер должен перейти в наш UI по диплинку, подключить кошелек и либо нажать там кнопку fund, ввести сумму депозита и подтвердить транзакцию, либо сразу после подключения кошелька отправить депозит (если в диплинк была зашита сумма депозита). Вместе с этим у юзера запросят signData для возможного деплоя других кошельков от этого агента (т.е. агент сможет создавать доп. кошельки без участия юзера). Таким образом сейчас флоу такой:
1. Пользователь находит Telegram-бота "DeDust Trader"
2. Бот: "Send me your TON address to get started", юзер отправляет адрес
3. Бот генерирует keypair и пишет "Wallet is ready. Enter the amount and currency of the first deposit" 
4. Юзер вводит сумму и валюту (например "10 TON")
5. Бот отправляет диплинк на наш дашборд, через который будет происходить деплой
6. Юзер переходит по диплинку
7. Юзер подключает кошелек 
8. Юзер отправляет транзакцию на пополнение кошелька (если используются intent transaction, то это происходит сразу на шаге 7)
9.  У пользователя гарантированно готовый задеплоенный кошелек, он возвращается в бота (по кнопке или вообще автоматически).
- **Всего 5-7 шагов от юзера, все очевидные и удобные**

На звонках предлагался так же флоу, когда юзер не деплоит agentic wallet самостоятельно и никуда не подключает свой кошелек, а агент просто скидывает юзеру адрес для пополнения. Я считаю, что этот подход ничем не лучше моего в плане удобства. Флоу при таком подходе такой:
1. Пользователь находит Telegram-бота "DeDust Trader"
2. Бот: "Send me your TON address to get started", пользователь отправляет адрес
3. Бот генерирует keypair и пишет "Wallet is ready. Address: UQ...Ac" 
4. Юзер **копирует** адрес пополнения (*не все знают, что моноширный шрифт копируется по клику*)
5. Юзер переходит в приложение кошелька (без диплинка, сам открывает приложение)
6. Юзер вставляет адрес пополнения
7. Юзер перепроверяет адрес пополнения (вдруг вставилось что-то не то)
8. Юзер выбирает валюту и вводит сумму пополнения
9. Юзер подтверждает транзакцию
10. Юзер возвращается в бота и надеется, что тот задеплоит кошелек. Затем юзеру все равно придется когда-то переходить на дашборд и подключать там кошелек, чтобы посмотреть кошельки агентов.
- **Всего 9 шагов от юзера, момент вставки кошелька не юзер-френдли**

То есть во втором варианте шагов даже больше, и они менее удобны для пользователя. При этом, во втором варианте есть проблема с деплоем кошелька. А в первом — нет. Единственное, возможно не получится газлесс отправить USDT для деплоя (этот момент требует уточнения у TonAPI)
